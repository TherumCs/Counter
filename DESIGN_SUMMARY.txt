KEY DESIGN DECISIONS
================================================================================

1. SCHEMA
   ✓ Leverage existing `position` columns on attributes, variants, images
   ✓ Add single generic `taxonomy_orders` table (v5) for hierarchies
   ✓ Parent/child relationships via self-referential FK (parent_id)
   ✓ Position column for sibling ordering
   Why: Avoids schema fragmentation, supports arbitrary custom taxonomies,
        coexists peacefully with existing position columns

2. DATA PERSISTENCE
   ✓ SQLite transactions for batch reorders (all-or-nothing)
   ✓ Position reflow only among affected siblings (no full table rewrites)
   ✓ Cycle detection before DB writes (validateNoCycles)
   ✓ Cascade delete: if parent deleted, children become root
   Why: Atomic guarantees, referential integrity, performance

3. ADMIN UI
   ✓ THREE separate pages (not one monolithic page):
     - Category Ordering (hierarchical)
     - Variant Options (flat or per-attribute)
     - Custom Taxonomies (generic for vendors, collections, etc.)
   ✓ Shared abstract base class for consistency
   ✓ SortableJS for drag-drop (no build step, 5KB gzipped)
   Why: Clear separation of concerns, reusable components, battle-tested library

4. REST API
   ✓ Single namespace: counter/v1/admin/taxonomies/{type}
   ✓ GET for listing (with optional include_tree=true for nesting)
   ✓ POST for batch reorder (all changes in one request)
   ✓ PUT/DELETE for single item ops (future enhancements)
   ✓ All endpoints check manage_woocommerce capability
   Why: Consistent with existing Counter REST patterns, atomic updates

5. FRONTEND RENDERING
   ✓ Serialize DOM tree on drag, validate before send
   ✓ Unsaved changes warning bar (yellow notice)
   ✓ Status feedback (saving…, saved!, error)
   ✓ Recursive tree rendering via JavaScript
   ✓ Keyboard nav support (Arrow keys, Tab, Enter)
   Why: No full page reload, immediate visual feedback, accessible

6. EXTENSIBILITY
   ✓ Custom taxonomy registration via filter hook
   ✓ Post-save action hook for cache invalidation
   ✓ Safe for third-party plugins to add ordering UI
   ✓ Backward compatible: all existing code still works unchanged
   Why: Counter philosophy of pluggability, no forced migrations

REST API ENDPOINTS
================================================================================

Authentication: All endpoints require manage_woocommerce capability

Endpoints:

1. GET /wp-json/counter/v1/admin/taxonomies/{type}
   Params: ?include_tree=true, ?parent_id=123
   Returns: { type, items: [...] } with optional nesting

2. POST /wp-json/counter/v1/admin/taxonomies/{type}/reorder
   Body: { moves: [{ id, parent_id, position }, ...] }
   Returns: { success, updated_count, tree }
   Validates: No cycles, parents exist
   Atomicity: SQLite transaction

3. PUT /wp-json/counter/v1/admin/taxonomies/{type}/{id}
   Body: { parent_id, position, enabled }
   Returns: Updated item

4. DELETE /wp-json/counter/v1/admin/taxonomies/{type}/{id}
   Returns: { deleted: true, id }
   Cascade: Children become root

DESIGN PRINCIPLES FOLLOWED
================================================================================

✓ Counter Architecture:
  - Service container (DI) pattern
  - Repository pattern for data access
  - REST controllers for API
  - Admin page classes for UI
  - PSR-4 namespace structure

✓ Counter Patterns:
  - Single namespace: counter/v1
  - manage_woocommerce capability check
  - wp_json_encode() for JSON
  - esc_html() for output
  - Transaction safety (SQLite BEGIN/COMMIT)

✓ Counter Conventions:
  - BEM CSS classes (counter-taxonomy-ordering__item)
  - Consistent color tokens (--counter-ac, --counter-ok)
  - Data attributes for JavaScript hooks ([data-taxonomy-list])
  - Font stack (JetBrains Mono for code, system font for UI)

✓ Accessibility (WCAG 2.1 AA):
  - Keyboard navigation (Arrow, Tab, Enter)
  - ARIA labels & live regions
  - Color + icon (not color alone)
  - Focus indicators visible
  - Screen reader announcements

✓ Security:
  - Parameterized SQL queries
  - Nonce verification on REST endpoints
  - Capability checks (manage_woocommerce)
  - Sanitized output (esc_html, wp_json_encode)
  - No arbitrary code execution

✓ Performance:
  - Single REST call for full tree
  - Batch reorder in single transaction
  - No N+1 queries
  - Minimal JS (SortableJS only)
  - Indexed database queries
